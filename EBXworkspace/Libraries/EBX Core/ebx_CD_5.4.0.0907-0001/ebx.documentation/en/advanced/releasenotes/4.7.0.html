Backward compatibility

A stricter access policy has been enforced in this version, in order to detect inconsistent accesses at the earliest stage:

  • The methods of class ProcedureContext performs additional checks so as to, for example, forbid transactions which span over several homes. If some existing client code does not abide by this requirement, it will henceforth throw an OperationException exception. We strongly recommend to correct such code, independently from adopting this new release: updating several homes in one transaction is a bad practice, which can - and will - cause inconsistencies in your Master Data.

  • Access to values through the methods Adaptation.getValueWithoutResolution or ValueContext.getValue could accept erroneous path in some cases. As for now, the existence and features of the specified node are systematically checked.

  • The class PrimaryKeyWildcard , deprecated since version 4.0.0, has been removed.

The new method Directory.getUsersInRole allows to retrieve all the users that are in the specified role. It is used by the new workflow engine. The default implementation of the method throws an exception. Consequently, if the workflow is activated and the environment is executing with a specific directory, the method Directory.getUsersInRole must be overridden.

User interface warning

  • The advanced input component has been activated by default. It can be disabled by means of the "ebxmanager-1.0" adaptation in the "Reference" branch.

  • Automatic hierarchy has been activated by default. It can be disabled by means of the "ebxmanager-1.0" adaptation in the "Reference" branch.

Workflow

Because Master Data needs more governance than any other data, workflow is a key feature for specifying, executing and monitoring the business processes that produce, update, merge and validate Master Data.

The new Workflow module focuses on three priorities:

  • Intuitive user interaction:

    Users get a clear view on the work they can participate to, and on the work they are committed to.

  • Reliability:

    Master Data must be managed in the safest possible way, hence it is essential that the MDM interaction be done in a secure, robust and declarative framework. The declarative features will ensure the explicit specification of processes and also reinforce a clear traceability of interactions.

  • Flexibility:

    Although apparently similar, business requirements on MDM processes are quite varied concerning the ordering of the tasks, the nature of individual tasks (automated versus user interactions), and their functional goal: global life-cycle management (branches, versions, and merges) or focused Master Data updates and validations. Additionally the scope and the expected result model of each task can be very specific.

For more information, read Introduction to workflow models.

In order to execute the workflow, please note that:

  • To activate the workflow, the property ebx.workflow.activation must be set in the configuration file 'ebx.properties'.

  • If the environment is executing with a specific directory, the method Directory.getUsersInRole must be overridden.

Inherited Attributes

As opposed to standard inheritance (which exploits a global built-in relationship between adaptation instances), inherited attributes are able to exploit finer-grained dependencies that are specific to the Master Data structure. It hence allows to factorize and adapt Master Data at business entities level.

For example, if the model specifies that a Product is associated to a FamilyOfProducts, it is possible that some attributes of Product inherit their value from attributes defined in its associated family.

For more information, read the section Inherited Attributes . 

Manager

Custom views and table sort

Sort criteria can now be precisely configured in custom views by means of a new user interface.

Additionally, the same user interface can be directly accessed through the "View..." menu of a default table view. Clicking on column header also remains available.

Duplication of perspectives

Custom views and Custom hierarchies can now be duplicated. New buttons "Duplicate this view" and "Duplicate this hierarchy" are displayed at the right of each custom perspective.

Table record form

The table record form header henceforth displays the record label as specified by osd:table/defaultLabel element.

CSV Exports and Imports

CSV imports are now available. This service is available in Manager, on a table page, in the "Actions..." menu. It allows to upload files from the client browser side.

CSV exports and imports now take into account the user-defined attributes (UDA) types, for elements with maxOccurs="1" . An UDA cell is displayed as "<attributeName>=<attributeValue>".

The CSV import can also be integrated programmatically, see the method ImportSpec.setCSVSpec .

Internationalization

Formatting policies

When a value is displayed to the user, it is formatted according to its type and the formatting policy of the current locale. For example, a date will be displayed in some languages as "dd/MM/yyyy". The per locale configuration of the formatting policy is now officially published.

For more information, read Formatting policies . 

Data Model Assistant

Various enhancements have been done, in order to render user interaction more intuitive, including the schema generation service.

Default labels of table are now managed.

XML Schema

Filters on foreign key constraints

The foreign key constraint, osd:tableRef , can henceforth declare a filter that selects the possible records of the target table. It can be done either by means of a parameterizable XPath expression, or through a specific JavaBean for complex computations.

For more information, read Foreign key constraint . 

Display of foreign key constraints

The foreign key constraint, osd:tableRef , can henceforth define more precisely how to display and sort the records of the target table. It can be done either by means of simple localizable pattern expressions, or by means of a specific JavaBean for more advanced display.

For more information, read Foreign key constraint . 

Table records labeling

The table declaration, osd:table , can define a specific JavaBean for advanced display of record, in the record page header.

It reuses the defaultLabel declaration, for more information, read Table Definition . 

API

Method Directory.getUsersInRole

The new method Directory.getUsersInRole allows to retrieve all the users that are in the specified role. It is used by the workflow engine to send emails when a work item is offered to a role.

The default implementation of the method throws an exception. Consequently, the method must be overridden if the environment has specified a specific directory.

Interface TableRefFilter

The new interface TableRefFilter allows to specify specific filters over foreign key constraints.

For more information, see TableRefFilter javadoc . 

Interface TableRefDisplay

The new interface TableRefDisplay allows to specify a specific display for a foreign key constraint (items' labels and sort).

For more information, see TableRefDisplay javadoc . 

Navigation link in UIBeanEditor

Methods have been added for getting a navigation link in a UIResponseContext , with the same signature as in the class ServiceContext ( getURLForSelection... ).

As a consequence, the class UIHttpNavigationLink has been deprecated.

For more information, see UIResponseContext javadoc . 

RequestResult.isSizeGreaterOrEqual(int)

The new method RequestResult.isSizeGreaterOrEqual(int) is provided as an addition to RequestResult.getSize() . It is intended as an optimization, for cases when the full computation of the request is not necessary.

For more information, see RequestResult javadoc . 

PrimaryKey factory methods

The javadoc of PrimaryKey class has been clarified and a new method AdaptationTable.computePrimaryKey(Object[]) facilitates the creation of a PrimaryKey from the primary key fields values.

Performance enhancements

Usage of existing indexes has been increased in order to speed up complex XPath requests. Among others, this can have a positive impact on selection nodes ( osd:select ) and uniqueness constraints ( xs:unique ).

Documentation and samples

In the developer reference part, a new chapter has been added in order to detail the various aspects of Inheritance and Value Resolution . 

In the developer reference part, a new chapter details of Labeling and Localization features.
