Workflow evolutions

"My work list" by publication

In Workflow Data, a new feature is available for visualizing the work items of the current user by publication. "My work list" is displayed when a publication is selected in the general work list. This feature is useful to take or open a work item directly, without selecting it in the general work list.

For more information, see My work list.

Enable confirmation request for decision

When the user saves his decision after the task execution by clicking on accept or reject button, a confirmation request is displayed. It is now possible to configure this confirmation request

For more information, see Enable confirmation request.

The script "Create a data space"

In a workflow model, in the script Create a data space , the new parameter branchNameForPermissionsCopy has been added for the permissions copy. Thanks to this parameter, it is now possible to define from which data space the permissions must be copied for the created data space. If no data space is defined for the copy, the parent data space is used by default.

Work item creation API

In a service extension, it is now possible to create a work item by specifying both an allocated user and a role for a possible reallocation. The work item is created in the allocated state, but an offered role is saved to make the allocation actions available.

For more information, see CreationWorkItemSpec.

Programmatic permissions rules

The workflow permissions have been contextualized and a new method is available to check the resolution context of the programmatic permissions rules (process instance or published process).

For more information, see ActionPermissionsOnWorkflow.

Data context API

A new method is available to check that the specified variable name is defined in the data context.

For more information, see DataContext.

Javadoc

The Javadoc about permissions rules, work item creation and associated contexts has been enriched.

Comparisons

With EBX5, every data comparison specifies whether it is in resolved mode. In previous versions, the resolved mode allowed to take inherited values into account. In this release, it also implies to take into account computed values (osd:function).

Consequently, comparisons performed by means of the API will now include differences between computed values: this affects the Java API invoked in resolved mode (see class DifferenceHelper) and the data service "Get changes" operation (it is always in resolved mode).

Another evolution is that now, within the user interface, the comparisons are all in resolved mode (previously, this was already the case for records comparison, but not for homes and instances comparisons).

Optimizations

Archives

The size of archives has been optimized, when they are exported with a change set and the export enables minimal content mode. In such cases, only records concerned by the change set are exported.

D3

The data transferred between a master and a slave have been optimized in order to be lighter and transmitted faster.

Version upgrade

Backward compatibility

Comparisons

The evolutions on comparisons can change the results when a comparison is made with the Java API in resolved mode or by means of the Data Service "Get Service" operation.

Validation rules

The interpretation of the validation rules has changed in this new version, in order to make their definition easier and more natural. Henceforth, a valid value has to comply with the criteria defined by a validation rule. This implies that all existing validation rules will become incorrect and must be updated in this way. That is, the existing validation rule should be converted to their negation for being valid regarding this new interpretation. See validation rules for more information.

Data model compilation

The data model compilation process is now stricter when defining a UILabelRenderer or a UILabelRendererForHierarchy on a table definition. At data model compilation time, an error message is now added to the compilation report if the Java class corresponding to a UILabelRenderer or a UILabelRendererForHierarchy cannot be found by the class loader.

For more information about UILabelRenderer and UILabelRendererForHierarchy , see UILabelRenderer and UILabelRendererForHierarchy . 

How to upgrade an installed environment

In order to upgrade your version of EBX5, perform the following steps:

  1. Backup your repository, including the RDBMS and the directory ebx.repository.directory.

  2. Replace the *.war and ebx.jar files on your application server.

Bug Fixes

Core Cache

  • [4149] On branch purge, the already purged branches should be deleted from the branches and versions technical table.

    If a branch has been purged from the database but is still present in the technical table, the branch purge throws an exception.

  • [4276] An error occurs when closing or re-opening a branch with a pipe char in the identifier.

    If a branch is created with a pipe char in the identifier, when closing or re-opening this branch, a NullPointerException occurs.

  • [4289] Error on Oracle if the database tables are not owned by the EBX applicative user.

    On some customer configurations, the instruction 'SELECT 1 FROM USER_TABLES' used by EBX to retrieve database tables will fail to see the existing tables (this is the case if a table exists, not in the current user schema, but through the declaration of a synonym).

  • [4298] Cannot create an instance, due to manual DDL scripts for repository creation being incomplete.

    The DDL scripts provided with EBX build 5.1.x did not include all the required updates. This causes the following error when creating any instance: Invalid object name 'EBX_XS'.

    This happens only if the scripts are run manually.

    Workaround: let EBX5 automatically install its tables.

  • [4299] In relational mode, changing primary key definition leads to an erroneous behavior.

    For a data model in relational mode, if the primary key definition is changed after the first publication, no error message is returned but the change is not correctly taken into account by the database.

  • [4345] Fatal error appears when entering values out of range ['-2147483648'; '2147483647'].

    A fatal error is displayed to the user when a custom views filter is applied if a value is out of range ['-2147483648'; '2147483647'] in its defintion

  • [4362]SchemaNode.getNode() can throw NullPointerException.

    In some case SchemaNode.getNode() can throw NullPointerException instead of returning null if no node exists at path.

Data Modeling

  • [4221] A erroneous data model can be published.

    A data model with an erroneous validation rule can be published. The result is a severe error on the resulting data model, and the relative contents become unavailable.

  • [4231] Data model import from a module cannot load the associated Java classes.

    A data model import from an existing module cannot load the associated Java resources.

  • [4233] Validation rules editor crashes on non-table record nodes.

    The validation rules editor is available on non-table record nodes ("instance nodes"), which is fine. However, editing the rules causes the assistant to crash (blank assistant screen and stack-trace in logs).

  • [4234] In the computation rule editor, the wizard cannot help on non-table record nodes.

    The computation rule editor is available on non-table record nodes ("instance nodes"), which is fine. However, the rules edition wizard cannot retrieve the other nodes to help in the rule design.

  • [4263] Specific error message on 'exclude segment' constraint is ignored at model publication.

    When a specific error message is defined in an 'exclude segment' constraint, a wrong element is generated at publication time in the target schema. Then, the specific error message is ignored in an associated data set.

  • [4310] Severity of a table validation rule is not set correctly when reverse-engineering a model.

    When a user imports an XSD model from the service "Import an existing data model", the severity of the validation rule is not set correctly.

    For example, if we have "warning" severity in the XSD file, after import it is defined "error".

Data Services

  • [4113] Auto-incremented fields should not be mandatory for the insert.

    In the WSDL, for the insert operation, the auto-incremented fields should not be mandatory, even if they are present in the XSD schema.

  • [4187] The label is not always optional in version creation data services.

    An exception occurs when executing the data service for version creation, when the label of the version is not defined whereas the description and the locale are defined. The service should be able to create a version without a label but with a description.

  • [4323] Unable to set a complex terminal node to null.

    There is no way to set a complex terminal node to null through web services. The assignment xsi:nil="true", which works on a simple terminal node, does not work here.

  • [4327] No option 'delta mode' exists in the generated WSDL files for update.

    The option delta mode, which guarantees the backward compatibility with V4,is not available in data services.

  • [4328] Non mandatory complex elements are not declared as nillable in WSDL.

    In order to allow the use of xsi:nil value, a non mandatory complex element must declare the attribute nillable="true".

Documentation

  • [4111] Web component: the URLs specified in the documentation are no longer valid.

    The documentation refers to services which have been moved.

Hierarchies

  • [4027] List field nodes should not be proposed at hierarchy label configuration.

    If the node is not a simple node with maxOccurs=1, the hierarchy configuration crashes. Multi-occurrenced field nodes should not be proposed at hierarchy label configuration.

  • [4117] In a hierarchy where node ordering is based on the join table dependency link, there may be duplicate child nodes.

    Since version 5.0.3, if a hierarchy dependency link is based on

    a join table, it is possible to configure as an ordering column,

    a column node from the join table instead of a column node from

    the target table.

    In this case, we may found duplicate nodes in the hierarchy displayed.

  • [4247] Join records linked to a deleted hierarchy member are not deleted.

    When deleting a hierarchy member with children linked by a join table, the join records are not deleted.

  • [4305] An exception occurs while computing the level of a hierarchy tree.

    If the data space referenced in a join table has been purged and

    no validation has been done to detect it, an exception is thrown.

Import Export

  • [4057] The order of exported columns is erroneous when exporting a table view via standard CSV export.

    The order of exported columns via standard CSV export is the order of the columns in the definition of the specific table view. It should be the order of the columns in the data model instead (this issue appears when the order in the data model is modified).

  • [4094] When importing a csv file, a field starting with the field delimiter character is not properly imported.

    When importing a csv file, if a field starts with the field delimiter character, this one is omitted. For example, the quote char results in an empty field after import.

  • [4098] A CSV file with UNIX line separators cannot be imported with the default import settings.

    A CSV file with UNIX line separators cannot be imported, as the line separators are not correctly detected. Explicit UNIX line separator can be programmatically set, but it is too technical an approach.

Miscellaneous

  • [3572] Unexpected behavior when a record that has been occulted or overwritten in a child data set is deleted.

    This issue applies to records that have been occulted or overwritten in a child data set.

    If the record is deleted, then when accessing the occulted or overwritten version in the child data set, the following behavior is observed:

    - If the record has been overwritten: the only fields holding a value are the primary key field(s) and the ones that have been overwritten. The other ones appear as "not defined".

    - If the record has been occulted: the only fields holding a value are the primary key field(s), all the other ones appear as "not defined". The "Inherit from parent DataSet" button still appears, and triggers an exception whenever clicked on.

  • [4294] Values set in a trigger are not shown after form submission.

    When the user submits a form for record creation, and the form includes a field whose value is set by the method TableTrigger.handleAfterCreate, then this value is not visible at first. The user has to click on the Revert button or refresh the page to see it.

User Interface

  • [3755] A pre-set value is not defined if the field is read-only.

    A value which is pre-set from an osd:select or a hierarchy is not set if the field is read-only.

  • [4101] Optimize the management of obsolete HTTP sessions.

    If many sub-sessions are created under the same HTTP session (for example, when viewing records involved in a merge), it is possible to invalidate a still-used root session.

  • [4172] The display of the comparison of two users in directory data set is erroneous.

    The display of the comparison of two users in directory data set is erroneous. The first column is compacted to the left and the GUI tries to display an encrypted password.

  • [4175] Record duplicate service is available even if the table is read-only.

    If the primary key of the table is an auto-increment, the duplicate record service is available even if the table has been defined as read-only in data set permissions.

  • [4178] Wrong behavior for integer input field when entering an out of range value ['-2147483648' ; '2147483647'].

    For integer fields, if the user fills in an out of range value ['-2147483648'; '2147483647'], he won't be able to select or edit this record anymore in spite of the fact that the value is accepted by EBX5.

  • [4225] Cannot select a record in hierarchy view, if not in first page.

    In the hierarchy view, in a target page distinct from the first one: when the user wants to see the detail of a record, the following error message is returned: "The element to be selected has not been found."

  • [4230] Filter button is too much on the right (firefox) / action button not align with object (chrome).

    In custom view creation in search criteria wizard filter button is too much on the right. It is also out of its frame.

    In dma action button can't align if expend/ collapse is used (must refresh to solve).

  • [4245] Data Spaces selector triggers NullPointerException.

    If the user has no preferred data space (last working data space), and access right to reference is forbidden, opening the data space selector throws a NullPointerException.

  • [4268] An exception occurs when creating a new record in a filtered table view.

    When accessing the table view filtered via an XPath request in the URI such as "/root/table[field='toto']", if a ValueFunction is defined on the field, an exception occurs when accessing the creation view.

  • [4274] With the computation rule editor, it is impossible to add a criterion under a logical block.

    In the Data Model Assistant, when using the computation rule editor, it is impossible to add a criterion under a logical block. The criterion creation pop-up gets opened but the button "Add" has no effect.

  • [4275] In the computation rules editor, "If true" and "If false" are never deleted.

    In the computation rules editor, "If true" and "If false" are not deleted along with the corresponding condition predicate.

  • [4303] When navigating through pages, page numbers are no longer displayed if the "last page" button is used.

    In the table view or hierarchy view, if the user clicks on the "last page" button, then the first record number and the total size are no longer displayed.

  • [4307] A fatal error occurs when applying a tabular view.

    A fatal error occurs when applying a tabular custom view.

  • [4324] In a newly created data space, the form does not display the updated values after a first submit.

    In a newly created data space, when users modify a record in an existing table and click on the "Submit" button, then the form displays the old values instead of the updated ones. Clicking on "Revert" displays the correct values, though.

  • [4336] Error when adding "Do not match any criteria" and "Do not match one of criteria" in Custom view.

    The system displays a fatal error when adding twice "Do not match any criteria" or "Do not match one of criteria" in Custom view.

  • [4337] Error Deletion in some tables.

    It is possible to create a record with special characters in some tables, but not to delete that record.

  • [4338] With IE6, wrong-sized interface checkbox in data space restriction for relational datamodel.

    The system displays too large a checkbox for data space restriction in relational datamodel

  • [4339] Wrong redirection when cancelling a CSV export that has been launched from a hierarchy.

    The cancel of a CSV export launched from a hierarchy drives the user to the content of the record instead of the hierarchy.

  • [4342] Refreshing schemas causes an invalid request.

    When viewing the validation report of a schema in the administration section: asking for a refresh of schemas causes an invalid request error message.

  • [4343] The services expand all and collapse all are unavailable on the root node.

    When there is no other service on a given node, the services expand all and collapse all are unavailable. As a side effect, these services are unavailable in Data modeling and Workflow modeling.

Users & Permissions

  • [4079] When the administrator has lost his built-in Role "Administrator", it cannot be recovered.

    Even with the property ebx.directory.factory=com.orchestranetworks.service.directory.DirectoryDefaultRecoverFactory, the administration rights cannot be recovered as described in the documentation "Administration Guide > Technical administration > Users and roles directory > Default directory > Administrator recovery procedure".

  • [4288] The dynamic permissions on workflow are not resolved timely, and get evaluated on terminated process instances.

    - If there is no process instance defined under a process publication, the permissions are not resolved and the method "ActionPermissionsOnWorkflow.getActionPermission" is never called.

    - Moreover, some permissions concerning only the publications are resolved for the process instances too.

    - When accessing the workflow tab, the dynamic permissions get evaluated on terminated process instances.

Validation

  • [4217] Validation rules do not return the expected result.

    A valid value is considered invalid, and conversely.

Workflow

  • [4193] The workflow sends empty notifications messages.

    When the template of the workflow notification message is not defined for the preferred locale of the notified user, empty messages are sent.

  • [4227] A condition task cannot modify the Data Context.

    Modifications of the Data Context within a Condition task are not taken into account.

  • [4246] An exception occurs when accessing a task definition.

    After a parameter of a script task has been deleted in the module.xml file, an exception occurs when accessing the definition of this task in the workflow steps tree.
