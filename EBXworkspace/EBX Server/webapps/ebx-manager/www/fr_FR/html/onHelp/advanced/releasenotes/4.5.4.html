4.5.4 fix 007

Release Date: February 13, 2009

(!) Upgrade of EBX.Platform from version 4.5.4 fix 006 to fix 007 must be considered.

Bug Fixes / Minor enhancement

  • [01829] Values set to null when creating a new table record are not preserved when EBX.Mananger redisplays same page because of a validation error.

  • [01832] After a schema modification which impacts the definition of a table primary key, accessing the table will fail (NullPointerException) because the persistent index does not reflect the modification. Note: this failure can be circumvented by manually deleting the existing index in filesystem.

  • [01833] In a filtered table accessed through "selection node", filter target nodes must be set to the filtered value and read-only, at table record creation (except creation by copy).

  • [01834] Searching a table record with a wrong structure of primary key(ex: through a wrong "piped" foreign key value) causes a fatal error.

  • [01835] If an instance is purged, then recreated with an identical reference, it will fail to be reloaded after server restart.

  • [01836] Data Services "GetChanges" operation should declare "minOccurs=0" in WDSL.

  • [01837]Adaptation.getTimeOfCreation() returns the date of creation of the instance, instead of the creation date of the table record.

Performance warning

  • This release adopts a new persistence format for table indexes (which are used to speed up access to table occurrences) to fix bug #01832. The index will be rebuilt on the first access to a table, which might result in load times longer than usual. Please note that this performance hit will only occur on the first access to a table. If you are concerned about this one-time performance hit, you can ensure a full index rebuild by validating the branch or the version.

New features

Hierarchies - Version 2

Enhancements:

  • Hierarchy nodes can be selected like records in table view.

  • A service can be executed directly from each hierarchy node by using the new contextual menu. This menu only provides services that are allowed according to permission rules.

  • Mass update is now available for hierarchy nodes.

  • Mass move is now available for hierarchy nodes.

  • The following "relationships" are now exploited for configuring a hierarchy:

    • osd:select elements,

    • inter-instance foreign keys ( osd:tableRef with container element defined).

Known limitations:

  • Other filters than views are not applied on hierarchies.

  • The perspective cannot be a combination of views and hierarchies.

  • Mass move under a recursive level is forbidden.

  • To avoid too much computing time load, dimension is limited to 10 levels depth and to 5 levels depth for very complex schemas containing more than 50 tables.

  • It is forbidden to move on a parent node which does not correspond to a terminal value.

Internationalization

  • The strategy has changed when no language is specified in module.xml : the default language is now en_US instead of fr_FR.

  • Schemas now support any languages, in addition to EBX.Platform languages.

  • The strategy has changed for FacetOResource. To be fully localizable, the resources must be at least declared in common directory and optionally overridden in each language matching directory.

Internationalization of enumerations

Labels associated to each xs:enumeration can henceforth be fully internationalized.

For more information, see section Enumeration labels . 

Cardinality constraint on selection nodes

The element maxOccurs has been added to the selection node specification ( osd:select ).

With maxOccurs , the validation checks that the contextual and dynamic selection is at most of the specified size. 

<xs:element name="linkToAuthorTitles" minOccurs="0" maxOccurs="0" > 
	<xs:annotation> 
	  <xs:appinfo> 
		<osd:select> 
		  <maxOccurs>10</maxOccurs> 
		  <xpath>//Titles[au_id =${../au_id}]</xpath> 
		</osd:select> 
	  </xs:appinfo> 
	</xs:annotation> 
</xs:element>

For more information, see section Tables, Filters and Selection Nodes . 

More powerful branch and version creation processes

The new class, HomeCreationSpec allows to create a branch or a version with specific features.

It is possible, for instance, to specify a source branch so as to copy its permissions to the new branch. Going further, a merge service can simulate a "logical branch" : the service first merges the branch to its parent, then it recreates a branch with the same properties, from the user point of view (only the internal identifier is changed).

For more information, see HomeCreationSpec javadoc . 

Other new functionalities:

  • Selected table's records can now be deleted (in table or hierarchy perspective).

    Note: This functionality is disabled by default. The proper permissions must be configured.

Performance enhancements

The display of adaptation tree is lighter so as to allow large trees (several hundreds of adaptation instances).

The records' selection in table and in hierarchies has been optimized.

Backward compatibility

  • Method Adaptation.getSchemaNode() : if the schema had errors, this method returned null . Henceforth, it throws UnavailableContentError , so that the client gets a clearer exception and for being consistent with the existing behaviour of other getter methods.
