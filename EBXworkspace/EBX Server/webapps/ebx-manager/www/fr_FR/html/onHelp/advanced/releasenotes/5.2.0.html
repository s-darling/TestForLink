User interface

Filters and search enhancements

The filters pane available on the table view has been enhanced with several new features: an advanced search filter, a more intuitive management of applied versus unapplied filters and a simplified but more useful text search.

The advanced search provides the ability to define precise criteria on-the-fly and to combine them logically. These filters can be saved for later use; once saved, they are displayed as a whole separate filter. The advanced search provides type-related operators and general operators, such as 'is null' and 'is not null'.

Each filter pane header now has a check-box: checking it applies the filter, unchecking it unapplies the filter.

The Text search has been simplified by removing useless options and by enhancing the search syntax:

  • it is possible to specify one of the usual wildcard characters, either the asterisk character ("*") or the question mark character ("?");

  • several words can be specified so as to be searched separately; they can also be enclosed in double quotes.

Multi-table views [beta]

It is henceforth possible to define multi-table views. Multi-table views are tabular views which involve several tables by performing table joins.

An easy-to-use wizard allows the definition of such views, offering the ability to exploit the relationships specified in the data model. This wizard can be accessed from the table view, menu "Views > Create a new custom view".

From the same "Views" menu, once a multi-table view is applied, it is possible to switch between the tabular presentation (information is displayed in rows and columns) and the hierarchical presentation.

Beta version: since this new feature is not yet fully optimized and all combinations are not yet supported, the creation of this type of view is disabled by default. To enable it, it is necessary to set the following property in the main configuration file (ebx.properties):

ebx.view.multitable.activated=true

Custom views unification

Three types of views are henceforth available: simple tabular, multi-table and hierarchies. These three types are all configured by means of a single wizard. Access to this wizard and to the available views is grouped under the same title "Custom views" in the "Views" menu.

This unification has also provided the opportunity to make the wizard more intuitive.

Permanent link

The user interface offers to copy the permanent link (or "permalink") of various entities. For this, the user has to go to the entity label, click on the question mark, thereby opening the entity's documentation pane, then click on the button on the upper-right of the documentation pane.

The permalink is available on data spaces, data sets, data set nodes, data set tables and records.

Confirmation before leaving a page

From now on, when the user modifies a form, then clicks on a link, the form will first ask for confirmation to leave the page.

Message box

Previously, messages of all severities (information, warning and error) were always displayed. With the new message box, only warnings and errors are displayed. An indicator on the top right of the workspace displays the number of unread messages for each severity. By clicking on it, the user is able to read the messages.

Table view sort indicators

The header of the table now displays all sorting arrows (on multiple columns). The sorting rank appears when the mouse rolls over the table header.

Documentation Search

Search in the documentation is henceforth available. A new field is displayed at the upper-right corner of the documentation window.

Internet Explorer 9 support

Microsoft Internet Explorer 9 is now officially supported.

Data modeling

Data model publication

The publication process has been enhanced: data models which do not depend on a module are embedded directly in the repository, without any XML Schema documents stored on the file system.

See Publication documentation for more information.

Data model versioning

It is now possible to manage several versions of a data model, compare them and perform a rollback to a previous version of a data model.

See Data model versioning documentation for more information.

Multiple publications

With the new publication management, it is possible to handle several data sets created from different versions of a data model. For that, distinct publications can be created from different versions of a data model.

See Multiple publications mode documentation for more information.

Data models export and import services

It is henceforth possible to upload ("Import XSD") and download ("Export XSD") XML Schema documents from the Internet browser. These services are available under the "Actions". The service previously named "Reverse" is also merged in the "Import XSD" service, to import XML Schema document from a module.

The "export archive" and "import archive" services have been moved to the "Manage versions" service.

History

Table history view

When the history has been activated on a table in the data model, it is henceforth possible to access the history view from various locations in the user interface: record, selection of records, table and data set.

In the table history view, permissions are resolved according to the current definition of permissions defined on the data set and on the data model. Table history views also benefit from standard functionalities such as exports, comparison and dedicated filters.

For more information, see table history view section.

Transaction history views

The new transaction history view gives access to the executed transactions, independently of a table, a data set or a data model, directly from the user interface.

A row of the transaction history displays the technical and authentication data of the transaction. As a transaction can perform several operations on a single table or on distinct tables and data sets, the impacted tables are also displayed with, for each table, the number of creations, updates and deletions. Table history views benefit from standard functionalities such as exports, comparison and dedicated filters.

For more information, see transaction history view section.

Aggregated lists

In the previous versions, it was not possible to historize a table containing an aggregated list (the data model reported an error). This restriction has been partially lifted, by allowing such cases; however, the corresponding fields will be ignored, namely not historized.

Workflow

Open the current work item from an email

The new workflow variable workflow.workItem.link has been added. It can be inserted in an email so as to open a web browser page on the current work item.

Also, a new field URL definition has been added in the mail configuration in Workflow-Executions. This field is used to build a concrete URL for the previous variable.

For more information, see the reference documentation.

New built-in condition

In the workflow model, a new built-in condition is available to check that a value is null or empty.

New built-in scripts compatible with internationalization

New scripts are available to provide internationalization compatibility and replace the old scripts "create a data space" and "create a snapshot". Old scripts are renamed with a "fr-en" suffix and deprecated. But the old scripts are still available to ensure backward compatibility.

Notification emails

In the workflow configuration, a new field allows a message to be declared as HTML in form.

Data services

Data set comparison

A new operation is available, it is now possible to compare two data sets of the same data model. The response will contain only the differences between tables.

Access to the default directory

The default user directory in EBX5 is implemented as a data set in a particular data space. In previous EBX5 versions, standard data services operations to access this data set were always forbidden; read-write operations are now allowed, provided that they are permitted by the standard permission system.

Java API

Integration of Ajax components

In order to allow finer and more dynamic user interactions, Ajax techniques can now be used in UI services and in UI beans.

For more information, see UIAjaxComponent documentation for more information.

Standard look and feel in UI services and UI components

With EBX5, specific user interface extensions and specializations are mostly done by means of the UI services and UI components. With this release, the Java API has been enhanced so that the developer can build more components complying with the standard EBX5 look and feel (layout, colors, fonts, etc.):

  • EBX5 table cell style (ability to customize background color and background image of a table cell, in a UI component)

  • Preview button (opens the standard modal inner pop-up)

  • Help button

  • JavaScript button (runs a JavaScript code)

  • Border (HTML fieldset)

  • Standard form row (composed of an icon, a label and the value display or component)

  • Standard form with buttons bar in UI services

  • basic HTML tables as CSS styles

  • Sub-titles and sub-sub-titles in UI services

  • "static" HTML tables

For UI services, the new interface UIServiceComponentWriter is available.

For UI components, the existing class UIResponseContext has been transformed to an interface and has new methods.

For basic CSS styles, common constants and static methods are available in the new classes UICSSClasses and UICSSUtils.

Customization of hyperlinks in hierarchy views

On hierarchy display, the default link on a hierarchy node can be overridden. For more information, see UILabelRendererForHierarchyContext documentation for more information.

History views related methods

New methods have been added to the public API in Adaptation and AdaptationTable.

See Adaptation.isHistory() and AdaptationTable.getHistory().

Miscellaneous

  • Getting the current directory from a user session is possible with the new method Session.getDirectory().

  • Defining a specific URL replacing the default login page can be done by overwrtiting the new method Directory.getUserAuthenticationURI.

  • The new method UITableFilter.hasFilterToApply() can be overridden to hide the check-box when the filter does not specify any criterion.

Version upgrade

Backward compatibility

Data model compilation

The data model compilation process is now stricter when defining several enumerations on a field. At data model compilation time, an error message is now added to the compilation report if several enumerations have been specified on a field.

Java API evolutions

Some UI classes (such as UIResponseContext) are now interfaces, when they formerly were abstract classes. This causes binary incompatibility between the 5.1 and 5.2 ebx.jar library and all existing client code which made use of these classes and compiled with a previous version. As a consequence, in order to avoid Java error IncompatibleClassChangeError, existing client code has to be rebuilt against the new 5.2 ebx.jar library, and redeployed together with these binaries.

Moreover, the refactoring done in the user interface extensions API has led to changes in the signature of the methods UIResponseContext.add...: the returned type has been "upcasted" (because these methods have been moved to either UIBodyWriter or UIJavaScriptWriter). In some cases, this could lead to compilation errors in existing client code.

Stricter behavior on JavaBean properties access

When a complex type definition is mapped to a JavaBean, each element under the complex type definition is mapped to the JavaBean property with a name identical to the element name. Moreover, the JavaBean property has to conform to the JavaBean protocol (usually by defining the corresponding getter and setter methods). In the previous EBX5 versions, if the read-access property threw an exception (for example because the getter method could not be accessed), this exception was masked and the method returned null. This behavior has been removed; henceforth, this can change the behavior of the application if a read-access property of a specific JavaBean throws an exception.

For more information on JavaBeans mapping, see Mapping of complex types to custom JavaBeans.

How to upgrade an installed environment

In order to upgrade your version of EBX5, perform the following steps:

  1. Backup your repository, including the RDBMS and the directory ebx.repository.directory.

  2. Replace the *.war and ebx.jar files on your application server.

Caution: This release contains new data modeling capabilities: if they are used, an existing repository will not be usable with previous versions of EBX5.

Bug Fixes

Data Modeling

  • [4280] When a UI service is defined on a data model with a servlet resource, validation errors occur.

    It is not possible to define a service referring to an HTTPServlet in the data model assistant.

User Interface

  • [3205] UIBeanEditor does not correctly handle an aggregated list if it is of a simple type.

    The editor for an aggregated list of a simple type does not behave in the same way as for an aggregated list of a complex type.

  • [3698] In the DMA view, clicking on the services arrow makes a scoll bar appear.

    In the DMA view, clicking on the services arrow makes a scoll bar appear. This is disturbing because we can no longer check the checkbox on the left.

  • [4315] Lack of a arrowhead in custom view and hierarchy menus.

    In the 'Views' menu for custom views and hierarchies, there is no arrowhead showing the existence of a sub-menu.

  • [4641] Waiting indicator always appears when importing a file.

    The waiting indicator is displayed as soon as the 'import XML' screen is displayed.

  • [4661] In Internet Explorer 7, quick help links do not work.

    The help window collapses when clicking on links.

Users & Permissions

  • [4683] If permission 'create data set' is disabled, the UI still displays "duplicate" action.

    Even when 'create data set' is disabled at data space level, the duplicate action is available (and cannot work).
